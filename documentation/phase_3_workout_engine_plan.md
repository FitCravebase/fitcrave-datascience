# FitCrave AI Backend: Phase 3 (Workout Engine) Implementation Plan

## Goal Description
The objective is to implement Phase 3 (Workout Engine) of the `fitcrave-datascience` backend as defined in the `fitcrave_ai_architecture.pdf`. The workout engine is responsible for generating periodized workout plans based on user goals/equipment, managing an exercise database, and tracking progressive overload using rule-based logic.

## Proposed Changes

### Configuration & Models
Models need to be accurately defined to handle the input and output of the workout engine.

#### [MODIFY] app/models/workout.py
- Define `Exercise` schema (name, muscle group, equipment, difficulty).
- Define `WorkoutSet` schema (reps, weight, RPE, rest_time).
- Define `WorkoutSession` schema (list of exercises with sets, cardio data, duration).
- Define `WorkoutPlan` schema (periodized weekly plan of sessions).

### Engine Logic
The core logic for plan generation, tracking, and database querying.

#### [NEW] app/engines/workout/data/exercises.json
- A mock/seed database containing a few sample exercises (e.g., Barbell Squat, Push-up, Dumbbell Row) to allow the engine to function and validate logic.

#### [MODIFY] app/engines/workout/exercise_db.py
- Implement logic to load `exercises.json` into memory.
- Implement search/filtering functions (e.g., `get_exercises_by_muscle_group`, `get_exercises_by_equipment`).

#### [MODIFY] app/engines/workout/plan_generator.py
- Implement an LLM call formatting function using `utils/llm_client.py` and `utils/prompts.py` to prompt Gemini 2.0 Flash to create a workout plan.
- The prompt will restrict the LLM to only select exercises from the allowed list generated by `exercise_db.py`.
- Enforce the LLM output to strictly match the `WorkoutPlan` Pydantic model.

#### [MODIFY] app/engines/workout/tracker.py
- Implement logic for logging a `WorkoutSession`. Note: This will primarily validate the input against the Pydantic schema and provide a stub for saving to the database.

#### [MODIFY] app/engines/workout/progressive_overload.py
- Implement rule-based logic to analyze a completed `WorkoutSession` against previous sessions.
- Example rule implementation: If in the previous session the target reps were met at the target RPE (Rate of Perceived Exertion), suggest a weight increase for the next session.

### Prompts
#### [MODIFY] app/utils/prompts.py
- Add prompt templates for `generate_workout_plan`.

## Verification Plan

### Automated Tests
I will create unit tests using `pytest` to verify the logic independently of the FastAPI server or real MongoDB connection.
1.  **Test Environment Setup:** Install `pytest` and `pytest-asyncio`.
2.  **`test_exercise_db.py`:** Test that `exercise_db.py` accurately loads JSON and filters exercises correctly.
3.  **`test_plan_generator.py`:** Test the structure of the LLM prompt and validate that a mocked LLM response successfully parses into a `WorkoutPlan` Pydantic model.
4.  **`test_progressive_overload.py`:** Test the rule-based logic with mock previous/current session data to ensure it correctly identifies when weight should increase or if volume needs to drop.

*Command to run tests:* 
```bash
pytest tests/
```

### Manual Verification
1.  Review code structure and Pydantic schemas.
2.  If the FastAPI server is running with routes exposed (later phase), use Swagger UI (`/docs`) to test generating a plan with mock user data (e.g., target muscle gain, 3 days/week, dumbbells only).
